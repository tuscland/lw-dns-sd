Installation

DNS-SD has no dependencies. Once downloaded, extract the files from the zip archive and load the file defsystem.lisp in LispWorks:

CL-USER 1 > (cd "~/lisp/dns-sd")
#P"/Users/camille/lisp/dns-sd/"

CL-USER 2 > (load "defsystem")
; Loading text file /Users/camille/lisp/plombier/dns-sd/defsystem.lisp
;; Looking at system "DNS-SD"
;; Looking at system "DNS-SD-TESTS"
#P"/Users/camille/lisp/dns-sd/defsystem.lisp"

You can then compile the system:

CL-USER 3 > (compile-system :dns-sd :load t)
;; compilation output...
(:DNS-SD)
This can also be done using the "Compile System" editor command.

Overview

Using DNS Service Discovery you will be able to:

  • Register, Browse and Resolve services on the network,
  • Get IP addresses from browsed hostnames,
  • Create, query, update and remove DNS resource records,
  • Create NAT port mappings, or get the NAT's external IP address.

Get an introduction on DNS Service Discovery.
This documentation was based on the C API Reference which can be found here.

The original C library is well covered, but with some limitations:

  • The following flags are not implemented at this moment:
    • kDNSServiceFlagsAllowRemoteQuery
    • kDNSServiceFlagsReturnIntermediates
    • kDNSServiceFlagsNonBrowsable
    • kDNSServiceFlagsShareConnection
    • kDNSServiceFlagsSuppressUnusable
    • kDNSServiceFlagsTimeout
    • kDNSServiceFlagsIncludeP2P
    • kDNSServiceFlagsWakeOnResolve
    • kDNSServiceFlagsBackgroundTrafficClass
    • kDNSServiceFlagsIncludeAWDL
  • The TXTRecord APIs are not implemented as foreign function calls because Lisp association-lists are used to build TXT records.

DNS-SD Operations

Functions in this section are similar to their respective counterparts in the DNS-SD C API.

[Function]
register port type &key interface-index name domain host txt-record no-auto-rename callback âž” operation

Schedules a service registration operation. The service is registered for as long as the operation has not been cancelled.

Signals a dns-sd-error if an error immediately occurs.

port specifies the port on which the service accepts connections. Pass 0 for a "placeholder" service (i.e. a service that will not be discovered by browsing, but will cause a name conflict if another client tries to register that same name). Most clients will not use placeholder services.

type is the service type followed by the protocol, separated by a dot (e.g. "_ftp._tcp"). The service type must be an underscore, followed by 1-15 characters, which may be letters, digits, or hyphens. The transport protocol must be "_tcp" or "_udp".

If not nil, interface-index specifies the interface on which to look for domains. (the index for a given interface is determined via the if-name-to-index family of calls.) Most applications will pass nil to register on all interfaces.

name, if non nil, specifies the service name to be registered. Most applications will not specify a name, in which case the computer name is used (this name is communicated to the client via the callback).

domain, if non nil, specifies the domain on which to advertise the service. Most applications will not specify a domain, instead automatically registering in the default domain(s).

host, if non nil, specifies the SRV target host name. Most applications will not specify a host, instead automatically using the machine's default host name(s). Note that specifying a non nil host does NOT create an address record for that host - the application is responsible for ensuring that the appropriate address record exists, or creating it via register-record.

txt-record specifies the txt-record associated with the service. It must be an association-list of key/value strings pairs.

no-auto-rename specifies renaming behavior on name conflict when registering non-shared records. By default, name conflicts are automatically handled by renaming the service. no-auto-rename overrides this behavior — with this flag set, name conflicts will result in a callback. The no-auto-rename option is only valid if a name is explicitly specified when registering the service (i.e. the default name, nil, is not used.)

When callback is specified, the operation is run asynchronously and results are delivered through the callback. callback must be a function of two arguments: operation and result where operation is the object returned by this function, and result is either a result, or an error if an error occurred in the dispatch process.
For register, the result object contains the following named values:

  • :PRESENCE — :ADD is used when the service has been successfully published, and :REMOVE when the service is about to be renamed in case of conflict resolution.
  • :NAME — the service name registered (if the application did not specify a name in register, this indicates what name was automatically chosen).
  • :TYPE — the type of service registered, as it was passed to the callout.
  • :DOMAIN — the domain on which the service was registered (if the application did not specify a domain in register, this indicates the default domain on which the service was registered).

See DNSServiceRegister.

[Function]
enumerate-domains &key interface-index domains callback âž” operation

Schedules a domains enumeration operation.

Signals a dns-sd-error if an error immediately occurs.

interface-index if not nil specifies the interface on which to look for domains. (the index for a given interface is determined via the if-name-to-index family of calls.) Most applications will pass nil to enumerate domains on all interfaces.

domains specifies the enumerated domains. The value can be :BROWSE-DOMAINS (the default value) or :REGISTRATION-DOMAINS.

When callback is specified, the operation is run asynchronously and results are delivered through the callback. callback must be a function of two arguments: operation and result where operation is the object returned by this function, and result is either a result, or an error if an error occurred in the dispatch process.
For enumerate-domains, the result object contains the following named values:

  • :PRESENCE — :ADD or :REMOVE depending on domains being found or lost.
  • :DEFAULTP — is t if the domain is the default domain.
  • :INTERFACE-INDEX — the interface on which the domain exists. (The index for a given interface is determined via the if-name-to-index family of calls.)
  • :DOMAIN — the name of the domain.

See DNSServiceEnumerateDomains.

[Function]
browse type &key interface-index domain callback âž” operation

Schedules a browse operation.

Signals a dns-sd-error if an error immediately occurs.

type is the service type being browsed for followed by the protocol, separated by a dot (e.g. "_ftp._tcp"). The transport protocol must be "_tcp" or "_udp". A client may optionally specify a single subtype to perform filtered browsing: e.g. browsing for "_primarytype._tcp,_subtype" will discover only those instances of "_primarytype._tcp" that were registered specifying "_subtype" in their list of registered subtypes.

If not nil, interface-index specifies the interface on which to look for domains. (the index for a given interface is determined via the if-name-to-index family of calls.) Most applications will pass nil to browse on all interfaces.

domain, if non nil, specifies the domain on which to advertise the service. Most applications will not specify a domain, instead automatically registering in the default domain(s).

When callback is specified, the operation is run asynchronously and results are delivered through the callback. callback must be a function of two arguments: operation and result where operation is the object returned by this function, and result is either a result, or an error if an error occurred in the dispatch process.
For browse, the result object contains the following named values:

  • :PRESENCE — :ADD or :REMOVE depending on services being found or lost.
  • :INTERFACE-INDEX — the interface on which the service is advertised. This index should be passed to resolve when resolving the service.
  • :NAME — the discovered service name. This name should be displayed to the user, and stored for subsequent use in the resolve call.
  • :TYPE — the service type, which is usually (but not always) the same as was passed to browse. One case where the discovered service type may not be the same as the requested service type is when using subtypes: The client may want to browse for only those ftp servers that allow anonymous connections. The client will pass the string "_ftp._tcp,_anon" to browse, but the type of the service that's discovered is simply "_ftp._tcp". The regtype for each discovered service instance should be stored along with the name, so that it can be passed to resolve when the service is later resolved.
  • :DOMAIN — the domain of the discovered service instance. This may or may not be the same as the domain that was passed to browse. The domain for each discovered service instance should be stored along with the name, so that it can be passed to resolve when the service is later resolved.

See DNSServiceBrowse.

[Function]
resolve name type domain &key interface-index broadcasting callback âž” operation

Schedules a resolve operation.

Signals a dns-sd-error if an error immediately occurs.

name, type and domain must be specified as the result values obtained from a browse operation.

interface-index specifies the interface on which to resolve a service. If this resolve call is as a result of a currently active browse operation, then the interface-index should be the index reported in the result. If this resolve call is using information previously saved (e.g. in a preference file) for later use, then use a nil interface-index, because the desired service may now be reachable via a different physical interface.

When broadcasting is :FORCE-MULTICAST the query is performed with a link-local mDNS query, even if the name is an apparently non-local name (i.e. a name not ending in ".local.").

When callback is specified, the operation is run asynchronously and results are delivered through the callback. callback must be a function of two arguments: operation and result where operation is the object returned by this function, and result is either a result, or an error if an error occurred in the dispatch process.
For resolve, the result object contains the following named values:

  • :INTERFACE-INDEX — the interface on which the service was resolved.
  • :FULL-NAME — the full service domain name, in the form <servicename>.<protocol>.<domain>.
  • :HOSTNAME — the target hostname of the machine providing the service.
  • :PORT — the port on which connections are accepted for this service.
  • :TXT-RECORD — the service's primary txt record, as an association-list of strings.

See DNSServiceResolve.

[Function]
get-addr-info hostname &key interface-index protocols broadcasting callback âž” operation

Schedules an IP address resolution operation.

Signals a dns-sd-error if an error immediately occurs.

hostname, specifies the fully qualified domain name of the host to be queried for, that should be the resulting value of a resolve operation.

interface-index specifies the interface on which to issue the query. Passing nil causes the query to be sent on all active interfaces via Multicast or the primary interface via Unicast.

protocols must be nil or a list of keywords containing :IPV4 to look up for IPv4 addresses, :IPV6 to look up for IPv6 addresses, or both keywords. If neither flag is set, the system will apply an intelligent heuristic, which is (currently) that it will attempt to look up both, except: if hostname is a wide-area unicast DNS hostname (i.e. not a ".local." name) but this host has no routable IPv6 address, then the call will not try to look up IPv6 addresses for hostname, since any addresses it found would be unlikely to be of any use anyway. Similarly, if this host has no routable IPv4 address, the call will not try to look up IPv4 addresses for hostname.

broadcasting can be :FORCE-MULTICAST or :LONG-LIVED-QUERY. Use :LONG-LIVED-QUERY to create a "long-lived" unicast query to a unicast DNS server that implements the protocol; this option has no effect on link-local multicast queries.

When callback is specified, the operation is run asynchronously and results are delivered through the callback. callback must be a function of two arguments: operation and result where operation is the object returned by this function, and result is either a result, or an error if an error occurred in the dispatch process.
For get-addr-info, the result object contains the following named values:

  • :PRESENCE — :ADD on successful address resolutions.
  • :INTERFACE-INDEX — the interface to which the answers pertain.
  • :HOSTNAME — the fully qualified domain name of the host to be queried for.
  • :ADDRESS — IPv4 or IPv6 address.
  • :TTL — if the client wishes to cache the result for performance reasons, the TTL indicates how long the client may legitimately hold onto this result, in seconds. After the TTL expires, the client should consider the result no longer valid, and if it requires this data again, it should be re-fetched with a new query. Of course, this only applies to clients that cancel the asynchronous operation when they get a result. Clients that leave the asynchronous operation running can safely assume that the data remains valid until they get another callback telling them otherwise.

See DNSServiceGetAddrInfo.

[Function]
nat-port-mapping-create internal-port &key interface-index external-port protocols ttl callback âž” operation

Schedules a NAT port mapping operation. The NAT port mapping exists for as long as the operation in pending (i.e. not cancelled).

Signals a dns-sd-error if an error immediately occurs.

internal-port specifies the port number on the local machine which is listening for packets.

interface-index specifies the interface on which to create port mappings in a NAT gateway. Passing nil causes the port mapping request to be sent on the primary interface.

external-port specifies the requested external port in the NAT gateway that you would like to map to the internal port. Pass 0 if you don't care which external port is chosen for you.

protocols must be nil or a list of keywords containing :UDP, :TCP, or both. The local listening port number must also be specified in the internal-port parameter. To just discover the NAT gateway's external IP address, pass 0 for internal-port, and leave the default value nil for protocols, external-port and ttl.

ttl specifies the requested renewal period of the NAT port mapping, in seconds. Most clients should use the default value.

When callback is specified, the operation is run asynchronously and results are delivered through the callback. callback must be a function of two arguments: operation and result where operation is the object returned by this function, and result is either a result, or an error if an error occurred in the dispatch process.
For nat-port-mapping-create, the result object contains the following named values:

  • :INTERFACE-INDEX — the interface through which the NAT gateway is reached.
  • :EXTERNAL-ADDRESS — IPv4 address string.
  • :PROTOCOLS — the list of protocols used : :UDP, :TCP or both.
  • :INTERNAL-PORT — the port on the local machine that was mapped.
  • :EXTERNAL-PORT — the actual external port in the NAT gateway that was mapped. This is likely to be different than the requested external port.
  • :TTL — the lifetime of the NAT port mapping created on the gateway. This controls how quickly stale mappings will be garbage-collected if the client machine crashes, suffers a power failure, is disconnected from the network, or suffers some other unfortunate demise which causes it to vanish without explicitly removing its NAT port mapping. It's possible that the ttl value will differ from the requested ttl value.

See DNSServiceNATPortMappingCreate.

[Function]
add-record operation rrtype rdata &key ttl âž” record-ref

Adds a records to a registered service. The name of the record will be the same as the registered service's name. The record can later be updated or deregistered by passing the record-ref returned by this function to update-record or remove-record.

Signals a dns-sd-error if an error immediately occurs.

operation must be a running operation returned by a call to register.

rrtype is a keyword representing the type of the record. Valid keywords are: :A :NS :MD :MF :CNAME :SOA :MB :MG :MR :NULL :WKS :PTR :HINFO :MINFO :MX :TXT :RP :AFSDB :X25 :ISDN :RT :NSAP :NSAP_PTR :SIG :KEY :PX :GPOS :AAAA :LOC :NXT :EID :NIMLOC :SRV :ATMA :NAPTR :KX :CERT :A6 :DNAME :SINK :OPT :APL :DS :SSHFP :IPSECKEY :RRSIG :NSEC :DNSKEY :DHCID :NSEC3 :NSEC3PARAM :HIP :SPF :UINFO :UID :GID :UNSPEC :TKEY :TSIG :IXFR :AXFR :MAILB :MAILA :ANY.

rdata is the raw rdata to be contained in the added resource record. It must be an array of (unsigned byte 8) elements. The contents of the data depends on the record type.

ttl specifies the time to live of the resource record, in seconds. Most clients should pass 0 to indicate that the system should select a sensible default value.

See DNSServiceAddRecord.

[Function]
update-record operation record-ref rdata &key ttl

Updates a registered resource record. The record must either be:

  • The primary TXT record of a service registered via register,
  • A record added to a registered service via add-record,

Signals a dns-sd-error if an error immediately occurs.

record-ref may be nil to update the primary TXT record, or the value returned by a call to add-record, or register-record.

rdata is the new raw rdata contained in the updated resource record. It must be an array of (unsigned byte 8) elements. The contents of the data depends on the record type.

ttl specifies the time to live of the resource record, in seconds. Most clients should pass 0 to indicate that the system should select a sensible default value.

See DNSServiceUpdateRecord.

[Function]
register-record operation full-name rrtype rdata &key interface-index rrclass identity ttl âž” record-ref

Registers resource records.

Signals a dns-sd-error if an error immediately occurs.

operation must be a running operation returned by a call to create-connection.

full-name specifies the full domain name of the resource record. The function construct-full-name can be used to format this string.

rrtype is a keyword representing the type of the record. rdata is the raw rdata to be contained in the registered resource record. It must be an array of (unsigned byte 8) elements. The contents of the data depends on the record type.

interface-index if non nil, specifies the interface on which to register the record (the index for a given interface is determined via the if-name-to-index family of calls.) Passing nil causes the record to be registered on all interfaces.

rrclass is the class of the resource record, currently supported value is :IN

identity when non nil, can have the keyword value :SHARED or :UNIQUE. :SHARED indicates that there may be multiple records with this name on the network (e.g. PTR records). :UNIQUE indicates that the record's name is to be unique on the network (e.g. SRV records).

Callback replies are made on the callback specified when the operation was created, using create-connection. For register-record, the result object contains the following named values:

  • :RECORD-REF — the record-ref object that can be used in other functions operating on records (e.g. update-record.

See DNSServiceRegisterRecord.

[Function]
reconfirm-record full-name rrtype rdata interface-index &key rrclass force

Instructs the daemon to verify the validity of a resource record that appears to be out of date (e.g. because TCP connection to a service's target failed.) Causes the record to be flushed from the daemon's cache (as well as all other daemons' caches on the network) if the record is determined to be invalid. Use this routine conservatively. Reconfirming a record necessarily consumes network bandwidth, so this should not be done indiscriminately.

Signals a dns-sd-error if an error immediately occurs.

full-name specifies the full domain name of the resource record. The function construct-full-name can be used to format this string.

rrtype is a keyword representing the type of the record. Valid keywords are: :A :NS :MD :MF :CNAME :SOA :MB :MG :MR :NULL :WKS :PTR :HINFO :MINFO :MX :TXT :RP :AFSDB :X25 :ISDN :RT :NSAP :NSAP_PTR :SIG :KEY :PX :GPOS :AAAA :LOC :NXT :EID :NIMLOC :SRV :ATMA :NAPTR :KX :CERT :A6 :DNAME :SINK :OPT :APL :DS :SSHFP :IPSECKEY :RRSIG :NSEC :DNSKEY :DHCID :NSEC3 :NSEC3PARAM :HIP :SPF :UINFO :UID :GID :UNSPEC :TKEY :TSIG :IXFR :AXFR :MAILB :MAILA :ANY.

rdata is the raw rdata of the resource record. It must be an array of (unsigned byte 8) elements. The contents of the data depends on the record type.

interface-index specifies the interface of the record in question. The caller must specify the interface. This API (by design) causes increased network traffic, so it requires the caller to be precise about which record should be reconfirmed. It is not possible to pass +interface-index-any+ for the interface index to perform a "wildcard" reconfirmation, where *all* matching records are reconfirmed.

rrclass is the class of the resource record, currently supported value is :IN

force pass t to force immediate deletion of record, instead of after some number of reconfirmation queries have gone unanswered.

See DNSServiceReconfirmRecord.

[Function]
wait-for-result operation &key timeout âž” result

Use this function to get the results of a running operation in a synchronous fashion. Returns a result if the call succeed in less that timeout seconds. If the result is a result-error that occurred within the dispatch loop, a call to wait-for-result will raise the error.

operation must be an operation returned by the DNS-SD Operations family of calls.

timeout is the time in seconds at which the function will raise a timeout-error. If nil is passed, the timeout is infinite (there is no timeout). The default value is *default-timeout*.

[Special variable]
*default-timeout*

Default value (60 seconds) for functions that requires a timeout keyword argument. nil means infinite wait.

[Function]
cancel operation &key callback timeout

Cancel a running operation.

operation must be a running operation.

With callback being nil, the call is synchronous, and operation is canceled after the call has returned. Otherwise, callback is a function of no argument called from the dispatch process when the operation is canceled, in which case the argument timeout is ignored.

timeout specifies the time in seconds after which the function signals a timeout-error. If nil, the wait is infinite. Default value is *cancel-timeout*.

Results

[Generic function]
result-more-coming-p result âž” boolean

Indicates that at least one more result is queued and will be delivered following immediately after this one. When the result-more-coming-p is t, applications should not immediately update their UI, because this can result in a great deal of ugly flickering on the screen, and can waste a great deal of CPU time repeatedly updating the screen with content that is then immediately erased, over and over. Applications should wait until result-more-coming-p is nil, and then update their UI when no more changes are imminent.

[Generic function]
